First: What You Are Designing (Mental Model)
You are not designing a “golf app.”
You are designing a live coordination surface that sits quietly between:
a complex, moving real-world schedule
and a group of people who do not want to think about it
Your job is not to impress users.
Your job is to disappear.
If someone notices your UI during the trip, you’ve probably failed.
The North Star Rule (Tattoo This)
The interface should answer questions faster than a human could ask them.
Everything you design either:
removes a question, or
creates one
Only the first is allowed.
Two Interfaces, Two Mindsets
1. Guest Command App
State of mind: distracted, outdoors, possibly drinking, phone in one hand, sunlight glare.
Design for:
glances, not reading
taps, not typing
certainty, not exploration
2. Operator Console
State of mind: focused, accountable, managing risk, moving pieces around.
Design for:
awareness
authority
consequence visibility
Do not try to make them feel the same.
Guest App: Design Instructions
Screen Hierarchy (Absolute)
NOW
NEXT
LATER
Everything else is secondary.
If the NOW screen fails, the app fails.
The NOW Screen (How to Design It)
This screen should feel like a departure board or a cockpit callout, not an app.
Layout rules:
One vertical column
Large typography
No scrolling
No icons unless they add clarity
NOW card
Dominant element on screen
Time is the largest text
Location is secondary
Action is obvious (map link)
This card should feel like a command, not information.
NEXT card
Smaller, but still legible
No actions besides awareness
LATER
Almost a whisper
Just enough to calm anxiety
Typography Rules (Guests)
Fewer weights, not more
Bigger text than you think
If it can’t be read at arm’s length in sunlight, it’s wrong
Avoid clever typefaces — neutrality wins
Color Usage (Guests)
Color is functional, not expressive.
Use color ONLY to indicate:
change
warning
confirmation
Most of the interface should feel monochrome.
If you want to add “fun,” do it with spacing, not color.
Interactions (Guests)
One tap should do one thing
Never stack actions
Never hide critical info behind gestures
No long presses
No tooltips
No tutorials
If a user needs to learn something, redesign it.
Live Scoring UI
This is not a game UI.
Design it like:
a paper scorecard
plus buttons
Rules:
Massive tap targets
No free-form text
Clear “saved” vs “synced” state
No animations when submitting scores
Silence is confidence.
Announcements & Messaging
This should feel like a bulletin board, not a chat.
Design cues:
Chronological
No avatars
No reactions
No reply affordances
This tells users subconsciously: you are not required to respond.
That reduces anxiety.
Help Button (Very Important)
This button should feel like calling the front desk, not opening a support chat.
Design it as:
calm
neutral
non-performative
No emojis.
No “We’re here for you!”
Just status and resolution.
Operator Console: Design Instructions
Overall Feel
This should feel like:
a flight operations system
or a show run-of-show board
Not a CRM.
Not a project management tool.
Dashboard Design
The dashboard is not a homepage.
It’s a risk surface.
Design it so that:
nothing looks urgent unless it is
issues surface quietly
the absence of alerts feels reassuring
Red should be rare.
Yellow should be thoughtful.
Green should be implied, not shown everywhere.
Timeline Builder (Most Important Screen)
You are designing cause and effect.
The operator must understand:
what is connected
what will break
who will be affected
Design requirements:
Blocks feel physical
Dependencies are visible
Editing a block shows consequences before committing
Never let an operator publish blindly.
“Impact Preview” UX
This is sacred.
When something changes:
Show exactly which guests will be affected
Show how their NOW/NEXT/LATER will change
Require explicit publish
This reinforces authority and responsibility.
Tournament Admin UI
This is not a fantasy sports product.
Design it like:
a rules sheet
plus a scoreboard
Transparency beats delight.
Issues / Incident Command
This UI should feel boring in the best way.
Lists
States
Ownership
Resolution
No dopamine.
No urgency theater.
Calm resolution builds trust.
Visual Language (Both Apps)
Shapes
Rectangles > rounded pills
Straight lines > playful curves
Shadows
Very light
Only for hierarchy, not decoration
Motion
State transitions only
Never celebrate
What You Are Actively Avoiding
Do not add:
personality
friendliness
jokes
“delight moments”
microcopy flair
Those belong in marketing, not in operations.
Final Design Test (Use This Every Time)
Before shipping any screen, ask:
Would a calm, competent human do this this way?
Does this reduce questions or create them?
Does this hold up if the user is distracted, late, or stressed?
Would this still exist if no one were watching a demo?
If any answer is “no,” cut or simplify.
Last Instruction (This Matters)
If the trip ends and no one says:
“That app was cool”
But instead says:
“That trip just worked”
You nailed it.
That’s the job.
