    * id, trip_id, owner, due_date, checklist, dependencies, status

The “Brain” part: rules + automation
You want the app to behave like a calm ops coordinator.
1) Constraint-based scheduling
Inputs:
* trip dates, tee time windows, travel time buffers, dinner preferences, activity blocks Rules:
* no overlaps
* buffers: 45–90 min before tee, 60–120 min after round before dinner
* “late arrivals” automatically placed in later groups Output:
* itinerary blocks + per-person personalized schedule
2) Shipping workflow with checklists
States:
* “invite sent” → “address confirmed” → “label created” → “in transit” → “delivered” → “received/staged” Auto:
* nag guests missing shipping info
* if tracking delays, trigger contingency plan: rental clubs / pro shop hold / expedited ship
3) Tournament generator
Choose:
* Ryder Cup, scramble, best ball, stableford, skins, wolf, etc. Auto:
* team balancing using handicap + “friend groups” constraint
* printable scorecards + rules page
* live leaderboard web view
4) Merch automation
Flow:
* collect sizes by deadline
* generate production-ready order (size matrix)
* assign kits to rooms or welcome table
* shipping timed to arrive before the group
5) Lodging & meals orchestration
* rooming assignment (bunk rules, snorers, couples, “don’t put these two together”)
* reservation tracking + confirmations
* dietary list exported for restaurants

The Real-Time App Concept
A mobile-first PWA (works like an app, no App Store friction) where every guest sees:
* Right now: next tee time / pickup / dinner / meetup point
* Live updates if anything shifts (weather delay, tee time change, shuttle moved, dinner moved)
* Tournament live scoring + leaderboard
* Group messaging + announcements
* Merch + team identity (teams, rules, formats, prizes, inside jokes)
* Help button (“my bag isn’t here”, “I’m late”, “where are we meeting”, “what’s the play?”)
Think: “Trip Captain” energy, but automated.

What makes it “real-time”
1) Live Itinerary Feed
Every schedule item is a “block” with:
* start time, location, map link, who it applies to, buffer rules
* status: upcoming / boarding / in-progress / complete / changed
When you edit one thing (tee time moved 20 min), the app:
* recalculates dependent blocks (shuttle pickup, warmup window, dinner buffer)
* sends a push notification only to impacted people
* updates everyone’s “Now / Next / Later” cards instantly
2) Live Scoring
* Guests enter scores hole-by-hole (or at the turn)
* Live leaderboard, match status, skins pot, etc.
* Rules baked in (handicap allowances, strokes per hole, formats)
3) Live Ops + Incident Handling
A small but crucial module: Issues
* “Bag delayed”
* “Rental clubs needed”
* “Someone’s missing”
* “Transport running late”
* “Dinner count changed”
Each issue has:
* owner (you / assistant)
* severity
* playbook (what to do next)
* chat thread and resolution log
This is where the app makes you look like a wizard.

The architecture that supports real-time
You want three layers:
A) Source of truth (database)
Postgres (Supabase) works fine.
B) Realtime delivery
* WebSockets / realtime channels for schedule + scoring + announcements
* Push notifications for “wake-up moments” (tee time moved, shuttle now, dinner check-in)
Supabase Realtime can do a lot here; you can also use something like Pusher/Ably later if you outgrow it.
C) Rules engine (the “brain”)
A service that:
* enforces constraints (buffers, travel time, dependencies)
* recomputes affected blocks when something changes
* triggers notifications based on impact
This is the thing that makes it feel intelligent instead of “a calendar”.
